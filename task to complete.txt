Full-Stack Engineering Task: Patient Context + Agent Workspace
Context
We are building DataX, a clinician-facing platform that helps clinicians work with patient context and medical knowledge faster.
This task is designed to evaluate your full-stack engineering ability, data modeling judgment, API design, clinician-facing UX, and ability to make safe trade-offs in a healthcare product.
Use only synthetic/mock patient data. Do not use real patient data.
Please do not spend more than 4–5 hours on this task.

Goal
Build a small runnable web application where a clinician can:
Create a synthetic patient.
View the patient record.
Retrieve structured patient context through an API.
Ask a bounded chat/agent interface to perform a task using that patient context.
Save a draft output with evidence references.
The goal is not to build a complete EHR or a perfect AI product. The goal is to show how you think about patient data, clinical context, APIs, agent workflows, and clinician-facing UX.

Mandatory Deliverables
Your submission must include:
A GitHub repository.
A browser-based web UI.
A backend service with APIs.
A README with setup instructions, API documentation, data model explanation, and trade-offs.
A locally runnable version of the app.

Required User Flow
The app should support the following flow:
1. Create Patient
Build a web form where a clinician can create a synthetic patient.
2. Patient List
Build a page where the clinician can see created patients and open a patient record.
3. Patient Detail Page
Build a patient detail page that shows the raw patient information in a clear, structured way.
The UI should distinguish between:
Raw patient data
Derived patient context
Agent/chat output
Saved draft output
4. Patient Context API
Build an API that returns structured patient context for a given patient.
This should not be a raw dump of the entire patient record.
The context API should return a purpose-specific clinical context object, such as:
Patient one-liner
Active problems
Current medications
Allergies
Relevant observations
Risk flags
Missing or unknown information
Evidence/source references
Limitations 
5. Chat / Agent Interface
Build a browser-based chat
The clinician should be able to ask the system to perform bounded tasks using the patient context.
Example supported requests:
“Prepare a handover summary for this patient.”
“What are the key risk flags?”
“What information is missing?”
“Summarize this patient for a new clinician taking over care.”
The agent response should show:
The task requested by the clinician
The patient context used
The steps the agent took
The generated draft output
Evidence/source references used
Missing information or uncertainty
A clear indication that the output is for clinician review only
The system should reject or safely redirect unsupported requests such as:
Final diagnosis
Treatment recommendation
Medication change recommendation
Anything not grounded in the available synthetic patient context
6. Save Draft
Allow the clinician to save the generated draft output.
The saved draft should preserve:
Patient ID
Draft title
Draft content
Task that created it
Evidence/source references used
Created timestamp
Required Data Model
Your implementation should show a thoughtful data model.
At minimum, your code or database schema should represent:
Patient
Conditions
Medications
Allergies
Observations / vitals / notes
Evidence or source references
Derived patient context
Agent task or chat message
Saved draft
You may use in-memory storage, SQLite, Postgres, or another simple local storage approach.
We care less about the database choice and more about whether the model separates raw patient data, derived context, agent output, evidence, and saved drafts.
LLM Usage
You may use an LLM for the chat/agent feature.
If you use an external LLM API:
Use only synthetic/mock data.
Make the API key configurable through environment variables.
Do not hardcode API keys.
Include a mock or fallback mode so we can review the app without providing our own API key.
We care more about the product and system design than the quality of LLM-generated text.
README Requirements
Your README must include:
How to run the project locally.
Tech stack used.
API documentation with example requests and responses.
Data model explanation.
Explanation of how the patient context is created.
Explanation of how the agent/chat workflow works.
What you intentionally kept out of scope.
Product or clinical safety risks you considered.
Trade-offs you made due to the time limit.
What you would improve if this became a real pilot.
What you would refuse to ship, even if asked.
